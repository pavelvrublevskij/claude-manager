Overview

What is Claude Manager?

A desktop-local web app that replaces manual editing of ~/.claude/ config files with a visual UI.

The Problem It Solves

Claude Code stores all its config in ~/.claude/ — JSON settings, Markdown memory files, JSONL sessions, MCP server definitions. Editing these by hand is error-prone and time-consuming.

The Solution

A single-page web app served at localhost:3000 that reads, validates, backs up, and writes those files through a clean UI — with zero cloud dependency.

Primary Users

Developers who use Claude Code daily and want visibility into token costs, session history, and configuration without leaving the browser.

Core Principle

Your data never leaves your machine. The server binds only to 127.0.0.1 and all writes are backed up automatically before any change.

Features — Part 1

Core Features

Configuration management, memory, and extension authoring.

Settings Editor

Visual tree editor for settings.json — permissions, hooks, env vars, API keys. Project-level and user-level scopes.

Memory Manager

Create, edit, and delete ~/.claude/memory/ Markdown files with frontmatter. Typed (user / feedback / project / reference).

CLAUDE.md Editor

Edit global and per-project instruction files with a Markdown editor. Changes take effect on next Claude Code session.

MCP Server Manager

Add, edit, enable/disable MCP servers across user, project, and local scopes. Supports SSE and stdio transports.

Skills (Slash Commands)

Author and manage custom /skill Markdown files. Previews rendered inline.

Output Styles & Agents

Define reusable response presets and project-level agent personas stored as Markdown.

Features — Part 2

Session & Terminal Features

Deep visibility into past conversations and an in-browser shell.

Browse all Claude Code sessions organized by project. Search messages, view tool-use results (Edit, Bash, Read, Write rendered structurally), rename sessions, and paginate large histories.

Security

Security & Safety Design

The app manages sensitive config — these safeguards are built into every write path.

  • ✓
    Localhost only — server binds to 127.0.0.1, never 0.0.0.0. Unreachable from network.
  • ✓
    Automatic backups — every write calls backup() first, storing a timestamped copy in ~/.claude/backups/.
  • ✓
    Path traversal prevention — safeSlug() and safeMemoryFile() validate all user-supplied paths before I/O.
  • ✓
    Read-only credentials — .credentials.json is displayed but never written by the app.
  • ✓
    No telemetry — zero outbound data. Pricing fetch is the only external HTTP call (public endpoint, no auth).
  • ✓
    No auth required — acceptable because the server is localhost-only and the machine's OS access controls apply.
  • ✓
    Standardized error handling — all route handlers wrapped in wrapRoute() — no unhandled rejections silently corrupting files.

Evaluation

Pros

What the project does well.

  • ✓
    Zero external dependencies at runtime — works fully offline; only npm packages needed, no cloud account.
  • ✓
    Extremely low barrier to entry — npm start and open a browser. No Docker, no database, no config.
  • ✓
    Single source of truth — reads directly from the real ~/.claude/ files Claude Code uses; no sync issues.
  • ✓
    Safety-first writes — automatic backups and path validation mean accidental damage is easy to undo.
  • ✓
    Cost visibility — token usage dashboard fills a real gap; Claude Code itself has no billing UI.
  • ✓
    Cross-platform — works on Windows, macOS, and Linux with platform-specific code paths where needed.
  • ✓
    No build step — contributors can edit a JS file and refresh the browser. Fast iteration cycle.
  • ✓
    Auto-update — one-click update from GitHub release ZIP without breaking local config.

Evaluation

Cons & Limitations

Trade-offs and areas for future improvement.

  • ✗
    Tightly coupled to Claude Code internals — file formats (~/.claude/ structure, JSONL schema) are undocumented and may change between Claude Code versions without notice.
  • ✗
    Global JS scope pollution — all modules are singletons on window. Naming collisions are a latent risk as the codebase grows.
  • ✗
    No TypeScript / type safety — the deliberate "no build step" constraint means runtime errors replace compile-time ones.
  • ✗
    node-pty native build — requires a native module compile on install, which can fail on machines without build tools (Python, MSBuild on Windows).
  • ⚠
    Large session files can be slow — JSONL parsing is done per request with no caching layer; very large conversation histories may cause noticeable latency.
  • ⚠
    No cross-session search — message search is per-session only; finding a specific snippet across all projects requires opening each session manually.
  • ⚠
    Manual startup required — the server must be launched explicitly each time; no background service or system tray integration.
  • ⚠
    Requires Node.js 18+ installed — a non-trivial prerequisite for users who only use Claude Code and don't otherwise develop in Node.
